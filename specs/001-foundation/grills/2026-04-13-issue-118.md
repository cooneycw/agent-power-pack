## Q1: Does the multi-runtime adapter architecture actually enforce coverage for all four first-class runtimes, or can a manifest slip through with partial runtime declarations?

The manifest validator (Pydantic schema in `src/agent_power_pack/manifest/`) enforces a `runtimes` field on every manifest YAML. The linter cross-checks that all four first-class runtimes (`claude-code`, `codex-cli`, `gemini-cli`, `cursor`) are declared. If a manifest omits any runtime, `make lint` fails and the PR is blocked. This is tested in `tests/unit/test_manifest_validator.py`. Gemini CLI and Cursor adapters are stubs for v0.1.0 — they install placeholder SKILL.md files — but the manifest schema still requires them, so no skill can ship without declaring all four. Constitutional Principle I is enforced at the schema level.

## Q2: Is AGENTS.md truly canonical, or could drift between AGENTS.md and generated files (CLAUDE.md, GEMINI.md, .cursorrules) go undetected?

The `agents-md:lint` phase-3 check (generated-file staleness) computes a hash of AGENTS.md content and compares it against the `<!-- GENERATED -->` header in each output file. If someone hand-edits CLAUDE.md, the linter detects the drift and either reverts it or fails. This is wired into both `make lint` and the `/flow:finish` gate — no PR can merge with stale generated files. The golden-file integration tests (`tests/integration/`) also verify that a round-trip from AGENTS.md through the generator produces byte-identical output. Constitutional Principle II holds.

## Q3: Does the single MCP container architecture handle concurrent multi-runtime access without port collisions or transport confusion?

The `compose.yaml` maps 12 ports: 8080–8085 for HTTP (stdio-style, used by Claude Code) and 9100–9105 for SSE/streamable HTTP (used by Codex CLI). Each of the six servers binds to exactly one HTTP port and one SSE port via `DualTransportRunner` in `supervisor.py`. The performance test `tests/perf/test_mcp_cold_start.py` asserts all six servers reach healthy within 15 seconds. The E2E smoke tests (`test_mcp_container_stdio.py`, `test_mcp_container_sse.py`) exercise both transports independently. The dual-attach test confirms concurrent Claude Code + Codex CLI sessions against the same container. Constitutional Principle III is validated.

## Q4: What happens if the grill-yourself threshold triggers during a trivial change that happens to touch a sensitive path (e.g., a one-line typo fix in AGENTS.md)?

The trigger logic in `src/agent_power_pack/grill/triggers.py` checks two independent conditions: (a) diff exceeds `max_lines` or `max_files` thresholds from `.specify/grill-triggers.yaml`, AND (b) a path-allowlist match (`mcp_container/`, `manifests/`, `adapters/`, `AGENTS.md`, `.specify/memory/constitution.md`). The path-allowlist fires regardless of diff size — so yes, a one-line AGENTS.md typo fix would trigger a grill. This is intentional: sensitive paths deserve scrutiny regardless of size. The user can suppress with `grill-yourself: skip` in the commit trailer if the change is genuinely trivial. The 26 test cases in `test_grill_triggers.py` cover this exact scenario (small diff + sensitive path = fires).

## Q5: Is the secrets tier system (dotenv → env-file → AWS sidecar) robust against partial failures, or could a missing sidecar silently return empty secrets?

The tiered resolution in `src/agent_power_pack/secrets/` probes each tier in order. If a tier raises an exception (e.g., sidecar HTTP timeout), it falls through to the next tier. If all tiers fail, it raises `SecretNotFoundError` — it never returns empty silently. The sidecar probe uses a health-check endpoint before attempting secret retrieval, so a down sidecar is detected early. Unit tests in `tests/unit/test_secrets_tiers.py` cover: dotenv-only, env-file fallback, sidecar fallback, and all-tiers-fail scenarios.

## Q6: Could the 60+ skill manifests across 13 families create maintenance burden? What prevents manifest rot?

The `agents-md:lint` repo-check phase validates that every Make target referenced in AGENTS.md actually exists in the Makefile, and every Docker service referenced exists in `compose.yaml`. The manifest validator ensures structural correctness. However, there is no automated check that ensures every manifest's `prompt` section stays current with the implementation. This is a known gap for v0.1.0 — the team accepts it because manifest prompts are treated as documentation (not executable code), and the skill catalog is regenerated on every `make install`. For v1.0.0, a prompt-drift detector is planned.

## Q7: Are the Gemini CLI and Cursor adapter stubs sufficient for the multi-runtime promise, or do they create a false sense of coverage?

The stubs install syntactically valid SKILL.md files with correct frontmatter but generic prompts. They pass `make lint` and `make install RUNTIME=gemini-cli` / `RUNTIME=cursor`. This is honest: the manifests declare all four runtimes, the adapters install files for all four, but only Claude Code and Codex CLI have been tested against real runtime sessions. The spec explicitly states Gemini and Cursor are "stubs for v0.1.0" and labels them P3. The integration tests use golden fixtures for all four adapters, so structural regressions would be caught even though functional regressions can't be tested without the real CLIs.
